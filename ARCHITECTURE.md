# Architecture

# Выбранный подход
В проекте выбрана монолитная архитектура.

# Почему выбран монолит
Для учебного проекта монолитное приложение является наиболее рациональным решением:
- проще разработка;
- быстрее запуск;
- меньше инфраструктурной сложности;
- легче отлаживать локально;
- достаточно для демонстрации полного цикла ML deployment.

# Состав сервиса
В текущей реализации приложение включает:
- код обучения модели;
- сохраненную модель;
- Flask API;
- Docker-контейнер для развертывания.

# Как это работает
1. Модель обучается на датасете UCI Credit Card.
2. Обученная модель сохраняется в `models/model.joblib`.
3. Flask API загружает модель при старте.
4. Endpoint `/predict` принимает признаки клиента и возвращает прогноз.
5. Endpoint `/health` проверяет работоспособность сервиса.

# Почему не микросервисы
Микросервисная архитектура здесь не выбрана, потому что:
- проект небольшой;
- нет высокой нагрузки;
- нет необходимости в независимом масштабировании частей системы;
- монолит проще сопровождать на этапе учебной разработки.

# Как можно развить систему в production
В production-версии можно выделить:
- отдельный inference-сервис;
- сервис логирования;
- систему мониторинга;
- брокер сообщений для асинхронной обработки;
- NGINX + Gunicorn/uWSGI для боевого развертывания.

# Концепция брокера сообщений
Если нагрузка увеличится, можно использовать RabbitMQ:
- API принимает запрос;
- сообщение с признаками отправляется в очередь;
- inference-worker читает очередь и делает предсказание;
- результат сохраняется или возвращается в downstream-систему.

## Логирование и мониторинг
Для production можно использовать:
- структурированные JSON-логи;
- ELK-стек для логирования;
- Prometheus + Grafana для мониторинга.

# MLOps-инструменты
В более зрелой реализации можно использовать:
- DVC—для контроля версий данных и артефактов;
- MLflow—для отслеживания экспериментов, метрик и версий моделей.
